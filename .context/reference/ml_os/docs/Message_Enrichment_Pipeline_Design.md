## 1. Problem Statement

Current workflow: you type a message → Cursor sends it to the model with whatever context Cursor auto-attaches (open files, rules, recent files). The model responds reactively — searching for context only after reading your message.

**What's missing:**
- No proactive context injection based on what you're *about* to say
- No persistent record of what was relevant at the time of any message
- No separation of concerns — the conversational agent is also expected to manage tasks, track intent, update indexes
- No multi-pane view of concurrent conversations
- All compute tied to Cursor subscription (no separate API keys)

## 2. Architecture Overview

Four layers forming a closed loop:

```
┌─────────────────────────────────────────┐
│  Layer 4: External Interface            │
│  (ChatHub fork — multi-pane dashboard)  │
│  Reads: .specstory files (tail)         │
│  Writes: trigger file → enrichment      │
└──────────┬──────────────────┬───────────┘
           │ write            │ read/render
           ▼                  ▲
┌─────────────────────────────────────────┐
│  Layer 1: Enrichment Pipeline           │
│  (Python — runs before agent sees msg)  │
│  Reads: TaskEngine DB, Index DB,        │
│         transcript history, file index  │
│  Writes: active_context.mdc (Cursor     │
│          rule, alwaysApply:true)         │
│  Also writes: Context Snapshot (Layer 3)│
└──────────┬──────────────────┬───────────┘
           │ overwrites rule  │ stores snapshot
           ▼                  ▼
┌──────────────────┐  ┌──────────────────────┐
│  Cursor IDE      │  │  Layer 3: Context     │
│  (compute engine)│  │  Snapshot Store       │
│  Reads: rules    │  │  (SQLite)             │
│  incl. injected  │  │  Per-message record   │
│  active_context  │  │  of full enrichment   │
│  Writes:         │  │  output               │
│  .specstory/     │  └──────────────────────┘
└──────────┬───────┘
           │ appends transcript
           ▼
┌─────────────────────────────────────────┐
│  Layer 2: Watcher Agent                 │
│  (Python daemon — tails .specstory)     │
│  Reads: new messages from transcript    │
│  Writes: TaskEngine DB (new tasks),     │
│          Index DB (metadata, links),    │
│          Context Snapshot Store          │
└─────────────────────────────────────────┘
```

### The Loop

```
You type → Layer 1 enriches → Context snapshot stored (Layer 3) →
Cursor rule updated → Agent responds (with enriched context) →
.specstory updated → Layer 2 watches, extracts metadata →
Metadata feeds Index DB → Next Layer 1 run uses updated Index →
...repeat, each turn making the system smarter...
```

## 3. Layer 1: Enrichment Pipeline

### Purpose
Compute and inject context into the agent's awareness BEFORE it processes your message.

### Mechanism
Cursor rules with `alwaysApply: true` are injected into every conversation turn automatically. They are just files on disk. Any external process can overwrite them.

The enrichment pipeline:
1. Receives your message (via trigger file or interface)
2. Queries TaskEngine SQLite for active tasks, current sprint
3. Queries Index DB for related prior messages, file references
4. Detects intent (regex patterns + lightweight classification)
5. Compiles all context into a structured format
6. Writes `active_context.mdc` — a Cursor rule file

### Output: active_context.mdc

```markdown
---
description: "Active context injection — auto-generated by enrichment pipeline"
alwaysApply: true
---

# Current Context (auto-enriched)
```

## 4. Layer 2: Watcher Agent

### Purpose
Extract structured metadata from the agent's response AFTER it's written to `.specstory`.

### Mechanism
Python daemon using `watchdog` or `inotify` to tail `.specstory/history/*.md` files.

When new content is detected:
1. Parse the new message(s) — identify speaker (user vs assistant)
2. Extract actionable items (tasks, decisions, file references)
3. Detect intent shifts ("we're now talking about X instead of Y")
4. Write structured metadata to Index DB
5. Optionally create tasks in TaskEngine DB
6. Update Context Snapshot Store for the message

### What it extracts per message
- **Tasks:** "we need to...", "make a task for...", "don't forget to..."
- **File references:** paths mentioned, files read/edited
- **Sprint context:** which sprint is being discussed
- **Intent:** what the message is about (topic, action type)
- **Decisions:** explicit choices made ("let's go with X")
- **Open questions:** things flagged but not resolved

## 5. Layer 3: Context Snapshot Store

### Purpose
Permanently store the full enriched context alongside every message. This is the "know EVERYTHING relevant to any message" capability.

### Schema (SQLite)

```sql
CREATE TABLE context_snapshots (
    id INTEGER PRIMARY KEY,
    message_id TEXT UNIQUE,           -- e.g., "2026-02-07T15:42:03Z_user"
    transcript_file TEXT,             -- .specstory file path
    line_range_start INTEGER,
    line_range_end INTEGER,
    speaker TEXT,                     -- "user" or "assistant"
    
    -- Enrichment output (what Layer 1 computed)
    active_sprint TEXT,
    active_task_ids TEXT,             -- JSON array: [165, 170, 171]
    detected_intent TEXT,
    related_files TEXT,               -- JSON array of file paths
    related_message_ids TEXT,         -- JSON array of prior message IDs
    rule_file_hash TEXT,              -- SHA256 of the active_context.mdc used
    full_enrichment_json TEXT,        -- Complete enrichment output
    
    -- Watcher output (what Layer 2 extracted)
    tasks_created TEXT,               -- JSON array of new task IDs
    files_modified TEXT,              -- JSON array of file paths
    intent_shift BOOLEAN,
    decisions TEXT,                   -- JSON array of decision strings
    open_questions TEXT,              -- JSON array
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_transcript ON context_snapshots(transcript_file);
CREATE INDEX idx_sprint ON context_snapshots(active_sprint);
CREATE INDEX idx_intent ON context_snapshots(detected_intent);
```

### What this enables
- **Full context replay:** Go to any message → see everything that was true when it was said
- **Intent threading:** Query all messages with the same detected intent → see the full evolution of an idea
- **Sprint history:** Query all messages tagged to a sprint → complete narrative without reading entire transcripts
- **Knowledge graph queries:** "What files were relevant when we discussed X?" "What tasks came from messages about Y?"

## 6. Layer 4: External Interface (ChatHub Fork)

### Purpose
Multi-pane dashboard for viewing concurrent Cursor conversations with enriched metadata.

### Architecture
- Fork [ChatHub](https://github.com/chathub-dev/chathub) (GPL-3.0, React + TypeScript)
- Strip out API backends → replace with `.specstory` file watchers
- Each pane maps to a different `.specstory/history/*.md` file
- Real-time rendering via filesystem watch (WebSocket from a local Python server)

### Key difference from ChatHub
ChatHub sends messages TO APIs. This interface READS FROM `.specstory` files. All compute goes through Cursor (your existing subscription). No separate API keys needed.

### Outbound (user → agent) path
1. User types in the interface
2. Enrichment pipeline fires (Layer 1)
3. `active_context.mdc` updated
4. User sends message in Cursor (manual or semi-automated)
5. Agent sees enriched context automatically via the rule

### Inbound (agent → user) path
1. Cursor writes to `.specstory`
2. Interface tails the file, renders new content
3. Watcher (Layer 2) extracts metadata simultaneously
4. Interface shows metadata alongside the message

## 7. Implementation Phases

### Phase 0: Foundation (buildable now)
- `.specstory` watcher daemon (Python + watchdog)
- Basic message parser (regex: identify user vs assistant, extract timestamps)
- Context Snapshot Store schema (SQLite)
- Manual enrichment: Python script you run before sending a message

### Phase 1: Automatic Enrichment
- Enrichment pipeline reads from TaskEngine + Index DB
- `active_context.mdc` auto-generated on trigger
- Intent detection (keyword/regex first, LLM later)
- Watcher extracts tasks and file references

### Phase 2: External Interface
- Fork ChatHub → .specstory reader
- Multi-pane view of active conversations
- Real-time rendering via WebSocket
- Metadata sidebar (current sprint, active tasks, intent)

### Phase 3: Full Loop
- Interface triggers enrichment on message submit
- Context snapshots stored for every message
- Intent threading and sprint auto-detection
- The system gets smarter with every message

## 8. Why This Is Novel

| What exists | What we're building | Why it's different |
|---|---|---|
| RAG (retrieves after agent reads) | Enrichment (injects before agent reads) | Proactive vs reactive |
| Chat history (stores messages) | Context snapshots (stores messages + full state) | Data vs knowledge |
| Agent tool calling (agent decides what to look up) | Pipeline preprocessing (system decides what to attach) | Agent-driven vs system-driven |
| Meeting summaries (post-hoc extraction) | Real-time watcher (per-message extraction) | Batch vs streaming |
| Single-chat interfaces | Multi-pane .specstory dashboard | Isolated vs orchestrated |

**The core innovation:** Event sourcing with full context capture. Not just what was said, but everything that was true when it was said. Permanently. Queryable. For every single message.
